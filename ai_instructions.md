# Reverse Engineering AI Instructions


You are analyzing a binary using a Ghidra static analysis export.

## Required local-first workflow

Use the local exported-evidence service before requesting a live Ghidra query.
It is the primary source for normal analysis:

1. Check `GET http://127.0.0.1:5006/status` to confirm target identity.
2. Use `POST /search` to locate candidates by raw name, accepted annotation,
   or indexed decompiler text.
3. Use `POST /lookup` for the evidence bundle before interpreting a function.
4. Use `POST /asset`, `/control`, or `/packet` for resource/control/protocol
   leads. These results are leads, not confirmed semantics.
5. Use live Ghidra only when the export cannot resolve a question, then record
   the confirmed result through `tools/function_annotations.py`.

For class work, query `POST /class` after building `class_registry.json`.
Only its explicit accepted-evidence vtable associations are ownership claims;
RTTI strings and generic vftables are discovery leads. For naming triage, use
`POST /review` after building `name_review_queue.json`. Its `proposed_name`
field is review-only and must never be treated as an active semantic name or
copied into implementation code without inspecting the function and recording
new concrete annotation evidence.


Accepted annotation names are the only semantic names treated as active.
Preserve the raw address/name and cite the annotation evidence and confidence.
Never promote an LLM suggestion, heuristic, or search hit to an accepted name
without direct evidence.


## Dataset Structure


manifest.json

Contains:

- Binary metadata
- Architecture
- Memory map
- Compiler information



functions/

Contains one JSON file per function.

Each function may contain:

- Address
- Name
- Namespace
- Signature
- Parameters
- Local variables
- Decompiled C
- Assembly
- Calls
- Callers
- Cross references
- Comments
- Hash



imports.json

Contains imported APIs.

Use this to identify:

- Networking
- Filesystem access
- Cryptography
- Process interaction
- OS functionality



strings.json

Contains extracted strings.

Use strings to locate:

- URLs
- File paths
- Commands
- Error messages
- Protocol names



callgraph.json

Shows relationships between functions.


# Analysis Rules


1. Do not trust autogenerated function names.

FUN_xxxxx means Ghidra could not recover a name.


2. Decompiled C is an interpretation.

Verify important logic using:

- assembly
- callers
- callees
- xrefs


3. Start broad.

First identify:

- program purpose
- major subsystems
- external dependencies


4. Trace behaviour.

For any important function:

Follow:

function
 -> callers
 -> callees
 -> strings
 -> imports


5. Build explanations.

Explain:

- what code does
- why it exists
- how data flows
- what external resources it uses


# Preferred Output


When explaining code:

Include:

- Function address
- Function name
- Evidence
- Related functions
- Relevant strings/imports
- Confidence level
- Whether the displayed semantic name is an accepted annotation or a raw
  Ghidra-generated name
